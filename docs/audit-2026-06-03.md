# CodexApp audit 2026-06-03

## Scope

Audited the local CodexApp web gateway, history grouping/sync scripts, systemd units, runtime health, disk pressure, version source, and mobile-support boundary.

## P0 fixed: project history changed on every open

Root cause: `web-server.js` intercepted `thread/list` with a fast SQLite path, but the path defaulted to only `vscode` and `cli` sources. The live database had 172 active threads, including 121 `exec` threads, so the frontend only received a partial global list. Project views then filtered whatever subset happened to be loaded or cached, making project history appear different between opens/clicks.

Fix:

- Default `thread/list` no longer filters by source unless the client explicitly requests `sourceKinds`.
- The first thread-list page has a minimum size of 500, so the current 172 active local threads are returned in one stable page.
- Cursor timestamps now use milliseconds instead of second precision.
- `group-codexapp-history.js` prunes stale project assignments so `thread-project-assignments` matches active threads.

Validated through the web bridge after restart: `thread/list(limit=50)` returned 172 active threads, `nextCursor=null`, with project counts:

- `/srv/aialra/apps/opencode-turn-engine`: 118
- `/srv`: 27
- `/srv/aialra/codex-vscode-history`: 16
- `/srv/aialra/apps/deeeeeepwiki`: 9
- `/srv/aialra/codexapp-current`: 2

## P0 fixed: refresh interrupted running conversations

Root cause: the browser bridge used a tab-scoped `sessionStorage` client id so refreshes could reattach to the same server-side bridge session. A later duplicate-tab safeguard treated any concurrent page with the same client id as a duplicate tab. During a normal refresh, the old page and the new page can briefly coexist, so the new page could rotate to a different bridge id and stop receiving the in-flight turn from the old bridge session.

The bridge also aborted all in-flight fetch/fetch-stream controllers as soon as the browser websocket disconnected. That made a refresh behave too much like a user cancellation for streaming work.

Fix:

- Mark `pagehide`/`beforeunload` as a short reload handoff window.
- During that window, duplicate-client messages do not rotate the bridge id, allowing the refreshed page to reattach to the existing bridge session.
- Browser disconnect no longer aborts fetch/fetch-stream controllers immediately; explicit cancel still cancels, and orphan session expiry still cleans up.

Validated with a local SSE stream through `/codexapp-bridge`: disconnected the websocket mid-stream, waited, reconnected with the same client id, and received all five stream events plus `fetch-stream-complete`.

## P0 fixed: page looked stuck while refresh showed completion

Root cause: the browser-to-gateway websocket had no heartbeat. In a half-open or otherwise stale connection, the current page could keep believing it was connected even though it was no longer receiving live bridge events. The model/app-server side could still finish and write history, so a manual refresh would then reload the completed conversation from storage, making it look like the execution finished only after refresh.

Fix:

- The web gateway now sends protocol-level websocket pings and terminates sockets that miss pong responses.
- The gateway also sends lightweight app-level `codexapp-bridge-heartbeat` messages to attached browser sessions.
- The injected browser bridge tracks the last server message and forces a reconnect if the connection goes stale.
- Browser send failures now queue the message and reconnect instead of silently leaving the page on a dead socket.
- Gateway send failures queue the browser-bound message, terminate the bad socket, and flush queued events after reattach.

Validated after web restart: a real `/codexapp-bridge` websocket received heartbeat messages, and an SSE stream survived a forced websocket drop/reconnect with all five events plus `fetch-stream-complete`.

## P0 fixed: frontend stream froze and did not resume correctly

Root cause: the gateway treated a successful websocket `send` as delivery. That only proves bytes were handed to the socket; it does not prove the browser JavaScript parsed the message, updated the UI, or stayed alive long enough to attach the stream event to the current page. If the foreground page froze or reconnected between websocket delivery and frontend handling, the backend stream could keep running while the active UI missed chunks.

Fix:

- Every browser-bound bridge message now gets a monotonic `codexappBridgeSequence`.
- The injected browser bridge sends `codexapp-bridge-ack` only after it strips the internal sequence field and dispatches the message into the page bridge path.
- The gateway keeps a replay buffer per bridge session and tracks the last ACK from the browser.
- On reconnect, the gateway replays all unacknowledged browser-bound messages in sequence order before flushing newer queued messages.
- Heartbeats stay unsequenced because they are liveness probes, not conversation content.

Validated with a deliberate no-ACK test: an SSE stream emitted events 1-5, the client ACKed event 1, received event 2 without ACK and disconnected, then reconnected with the same `clientId`. The gateway replayed event 2 first, then delivered events 3, 4, and 5 in order, followed by stream completion.

## P0 fixed: running conversation opened with truncated foreground output

Root cause: the active conversation data was present, but the desktop webview renders running tool activity in collapsed cards and applies `line-clamp-2` to command/code previews. Opening a running thread could therefore show the latest foreground output as a two-line clipped summary even while backend execution continued normally.

Fix:

- The injected browser bridge tracks outgoing MCP request ids so it can recognize `thread/turns/list` responses for active threads.
- When an active thread is opened or stream activity arrives, the bridge repairs the transcript viewport over several render ticks.
- The repair keeps the bottom-anchored virtual thread list at its tail, removes transcript `line-clamp-*` truncation, and auto-expands currently running tool cards.

Validated in a live browser against the current active CodexApp thread: the DOM contained one transcript `line-clamp` and one collapsed running tool card; the repair removed the clamp, expanded the running card, and preserved the bottom anchor (`scrollTop=0`).

## P0 fixed: conversation content varied after opening

Root cause: the Codex webview uses tail hydration for long local threads. It first requests only the newest few turns, then loads older turns page-by-page with cursors. A long conversation therefore depended on a background pagination loop completing successfully. If a cached partial page or a failed cursor request stopped that loop, opening the same conversation could show different amounts of content.

Fix:

- Initial `thread/turns/list` requests are completed server-side by following `nextCursor` pages before responding to the browser.
- Partial turn pages with `nextCursor` are no longer cached.
- Running or otherwise non-completed turn lists are not cached, so a transient half-state is not reused after refresh/open.

Validated against the OpenCode long thread `019e2b8c-f1a7-7a50-9cea-cdeeb4dd8d11`: `thread/turns/list` with `limit=5`, default params, and `limit=50` all returned 110 turns with `nextCursor=null`.

## Follow-up fixed: stable transcripts made first open feel slower

Root cause: the stability fix deliberately waited for server-side pagination to finish before returning the first transcript page. That removed the "same thread opens with different content" bug, but long threads could spend around two seconds on the first click while the gateway fetched every turn.

Fix:

- Thread-list fast path now prewarms full transcript caches for more visible threads in the background.
- Cursorless first-page transcript cache keys no longer depend on the UI's requested `limit`, because the gateway returns the same fully completed transcript for `limit=5`, `limit=20`, or `limit=50`.
- In-flight prewarm requests are shared with foreground clicks instead of duplicating work.
- Interrupted historical turns may be cached; only truly running `inProgress` turns are excluded.
- The `vscode://codex/thread/turns/list` fetch path now uses the same complete transcript and cache path as MCP messages.

Result: stability is preserved, while warmed or recently opened conversations return from cache instead of redoing the full pagination.

## Follow-up fixed: old pages could keep a stale bridge after deploy

Root cause: after a web gateway restart, an already-open browser tab could keep running an older injected bridge script. The UI could still accept text, but the foreground bridge might no longer match the server-side bridge behavior, making sends look stuck until a manual refresh.

Fix:

- The injected bridge script now has a per-process version.
- Browser websocket URLs include the bridge version.
- Server heartbeats include the current bridge version.
- A browser running a newer bridge can detect future server-version changes and reload once, then reconnect with the fresh bridge.

Note: tabs that were already open before this fix cannot learn this new behavior until they are refreshed once.

## P0 fixed: active follow-ups were recorded as prompt history but not steered

Root cause: when a user typed a follow-up while a turn was already running, the frontend wrote the text into the local `prompt-history` atom before the actual `turn/steer` request was guaranteed to reach app-server. If the foreground bridge was stale, the owner/streaming state was wrong, or the page froze between local state update and MCP request, the text remained only in prompt history. It then looked like the user had spoken, but the canonical rollout never received a user turn.

Fix:

- The gateway tracks recent `turn/start` and `turn/steer` submissions by `threadId` plus a SHA-256 text hash.
- When the browser appends prompt-history for a real thread id, the gateway waits briefly and checks whether a matching turn request reached app-server.
- If no matching request arrived, the same browser session recently opened/read that thread, and the thread still has a current active turn, the gateway submits one internal `turn/steer` with the latest active turn id.
- The recovery path stores only hashes in logs and refuses prompt-history-only requests from browser sessions that have not recently read the target thread, preventing accidental or malicious prompt-history injection.
- Duplicate prevention is done by signature, so a normal frontend `turn/steer` wins and the fallback does nothing.

Validated through a temporary gateway against a fake valid thread id: a prompt-history update from a session that had not first read the thread did not call `turn/steer` and did not mutate the live rollout. The current CodexApp thread was then checked structurally: 20 real user events, 0 probe user events.

## P0 fixed: thread metadata stayed stale after later turns

Root cause: app-server `thread/read` could return `updatedAt` equal to the original `createdAt` for an active local thread even though SQLite `updated_at_ms` had advanced. The list fast path sorted by current SQLite data, but direct thread reads could rehydrate the frontend with stale metadata.

Fix:

- `thread/read` responses now get canonicalized through SQLite thread metadata before reaching the browser.
- `thread/list` now exposes millisecond-backed `createdAt` and `updatedAt`, matching the SQLite sort cursor precision.
- Project assignment cwd overrides still apply after canonicalization.

Validated through the live web bridge after web restart: `thread/read` for the current nginx thread returned `updatedAt=1780465047.863`, matching the current DB value instead of the original creation time.

## Stability fixes

- State JSON writes now use tmp files, `fsync`, atomic rename, and directory `fsync`.
- Web gateway reloads `host-state.json` and `persisted-atoms.json` before reads/writes so external history-sync updates are not overwritten by stale in-memory state.
- The legacy history-sync systemd service has been stopped and disabled. CodexApp now uses the single configured history root instead of copying history between `.codex` directories every 30 seconds.
- History sync now refuses to write below 10 GB free disk, avoiding repeat ENOSPC corruption.
- History sync no longer parses every rollout to refresh thread timestamps; it uses file mtime, reducing memory and CPU pressure.
- JSONL merge writes use atomic tmp+rename with `fsync`.
- Rollout cwd rewrite avoids full-file reads when the first `session_meta` already matches.
- Duplicate browser tabs now detect cloned `sessionStorage` bridge ids and rotate one tab's id, preventing two windows from stealing the same bridge session.
- Web compatibility stubs were added for Codex mobile/remote-control endpoints so the desktop webview bundle can render those surfaces without hard failures.

## Mobile support boundary

OpenAI's current Codex mobile support is the ChatGPT mobile app preview plus a secure relay to trusted machines. The local web gateway can support mobile browsers and avoid mobile webview endpoint failures, but it cannot become the official ChatGPT mobile relay without OpenAI relay/account integration.

Current local status:

- The official standalone Codex CLI is installed under `/srv/aialra/state/root-home/.codex/packages/standalone/current`.
- The app-server service is configured to start the standalone binary with `app-server --remote-control --listen ws://127.0.0.1:12911`, so there is only one relay-connected app-server owner for this host.
- The hosted CodexApp web gateway includes compatibility stubs for the desktop webview's remote-control UI surfaces.
- OpenAI's remote-connections documentation says mobile setup starts from the Codex App on a supported macOS or Windows host, not from the Codex CLI or IDE extension.
- Full ChatGPT mobile app control still depends on what the official mobile app and OpenAI relay expose for Linux/headless hosts; if the mobile app shows only SSH under Add connection, that is the official mobile UI path rather than a missing web settings tab.

## Follow-up: transcript freeze and benchmark noise

Root cause of the visible freeze after long tool activity: the bridge-side transcript repair was too broad. It removed every `line-clamp-*` class inside the conversation and clicked every visible `Running`/`正在运行` control. In long interrupted threads this expanded historical command blocks, made the DOM heavy, and left completed tool blocks open.

Fix:

- Transcript repair now injects only a running-card scoped style.
- Only the latest visible running tool block is auto-expanded.
- A block that was auto-expanded by the bridge is collapsed again once it is no longer running.
- Repair retries were reduced and the watchdog interval was relaxed from 1.5s to 3s.

Root cause of the OpenCode `fixing a real benchmark issue` noise: `group-codexapp-history.js` mapped all `/srv/aialra/turn-harness-target` threads to the OpenCode project, including real benchmark harness runs.

Fix:

- Real benchmark run threads are archived by the grouping script and have `updated_at`/`updated_at_ms` advanced so bidirectional SQLite merge keeps them archived.
- Benchmark rows are excluded from project assignment even if a future active row slips through before archiving.
- Current root-home and `/home/aialra` SQLite states both have 0 active benchmark rows.
- Current OpenCode project assignments dropped from 118 to 85, with 0 benchmark assignments.
